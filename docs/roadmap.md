# ロードマップ — sabotage(サボタージュ)

## データソース

### ThemeParks.wiki API(メイン・キー不要)
- ベース: `https://api.themeparks.wiki/v1`
- `GET /destinations` — 全パーク一覧。東京ディズニーシーのentity IDは実行時に発見してキャッシュする
- `GET /entity/{id}/live` — パーク配下全エンティティのライブデータ(待ち時間・運営状況・ショー時刻)
- `GET /entity/{id}/schedule` — 開園スケジュール
- 東京ディズニーランド entity ID: `3cc919f1-d16d-43e0-8c3f-1dd269bd1a42`
- 注意: 実装前に必ず実APIを叩いてレスポンス形状を確認し、その実形状を
  テスト用フィクスチャに使うこと。フィールド名の決め打ち禁止

### Queue-Times API(サブ・突合と冗長化用)
- `GET https://queue-times.com/parks.json` — 東京2パークのIDを実行時に発見
- `GET https://queue-times.com/parks/{id}/queue_times.json` — 待ち時間(5分ごと更新)
- クレジット表記必須(CLAUDE.md参照)。is_openしか無いためDOWN/CLOSED非区別。
  停止検知はThemeParks.wikiを正とする

### 天気(人流再配分イベントの予告)
- Open-Meteo(キー不要)で舞浜(緯度35.63, 経度139.88)の降水予測を取得(Phase 2で導入)
- **実装済み**(`data/weather.py`):気温・降水・**降水確率(今以降2時間の最大)**を
  ポーラー/毎時フライホイールが待ち時間と一緒に `weather` テーブルへ記録。現況ボードに
  気温バッジと「まもなく雨→室内混雑注意」を表示。降水確率の立ち上がりが屋内退避の先行シグナル。

## Phase 0: データフライホイール(最優先)
ポーラー + SQLite蓄積。すべての後続フェーズの燃料。
- 5分間隔+ジッターで両パークの /live を取得(cron向けに --once モードを用意)
- テーブル:
  - `snapshots(ts, source, park_id, http_status, raw_json)` — 生レスポンスを丸ごと保存
    (正規化スキーマの設計ミスから常に復旧できるように)
  - `observations(ts, park_id, entity_id, name, entity_type, status, wait_minutes)` — 正規化済み
  - `meta(key, value)` — 発見したパークIDのキャッシュ
- 取得失敗・欠測も status='FETCH_FAILED' の1行として記録(欠測は観測である)
- SQLiteはWALモード。1サイクル内の例外でループが死なないこと

**DoD**: 24時間無人で回り続け、欠測が欠測として残る。再起動しても壊れない。
テストはネットワーク非依存(実形状のフィクスチャ駆動)で通る。

## Phase 2: 人流ナウキャスティング
待ち時間は遅行指標。先行指標から10〜20分先を予測する。
- ショー/パレード終了時刻(snapshots内showtimes)→ 周辺への衝撃波モデル
- 降水予測 → 屋内退避の再配分イベント検知
- 待ち時間フェアバリュー: 曜日×時間帯の期待値カーブと現在値の乖離(割安/割高)
- **群衆弾性の推定**: アトラクション別「急落が埋まる速度」(dipの平均回帰速度)を
  蓄積データから推定し、到着時待ち時間の予測(herd_adjusted_wait)を実装する
- **機会発生密度マップ**: 急落・運転再開が起きやすい場所×時間帯の分布(「網」を張る位置の根拠)

**DoD**: 過去データでバックテストし「スパイク10分前検知」の的中率を数字で出す。

## Phase 3: 行程国会(マルチエージェント討論)
討論で行程ポリシーを生成。Anthropic API(モデルは claude-sonnet-4-6 目安)。
- エージェント: 乗り物至上主義 / 飯 / だらだら原理主義 / 敵対(脆弱性を刺す)/ 鯤枠(偶然との交換提案権)
- 出力は時系列の「順序」ではなく「ポリシーJSON」: 条件→行動のルール集合
- 審査基準:
  - ピークエンド: ピーク1つと締め1つが確保されていない計画は不採択
  - 決断関数適合: すべての移動提案が V_go/V_near/V_stay の形式で正当化されていること
    (同点は常にstay — CLAUDE.md 心理設計原則2)
  - ユリシーズ契約: すべての移動提案に撤退条件(abort_if_wait_exceeds)が付いていること

**DoD**: 日付と好みを入れると、討論ログ付きでポリシーJSONを出力するCLI。

## Phase 4: BCPポリシーエンジン
- 蓄積データからアトラクション別の停止頻度テーブルを生成
- ポリシーJSONを解釈し、DOWN検知・降雨でフォールバック分岐を自動発動(木鶏の実装)
- 通知はまず標準出力/ログで十分

**DoD**: 過去の任意の1日をリプレイして、停止イベントでポリシーが正しく分岐する。
